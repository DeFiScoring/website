Audit this project’s secrets usage and migrate secrets to the right home: GitHub Secrets for build/deploy automation, Cloudflare secrets/environment variables for runtime. Choose whichever makes the most sense per secret, not one-size-fits-all.

Goals:
1. Find every secret currently used in the repo, Replit Secrets, env files, docs, scripts, CI, and deployment config.
2. Classify each secret by use:
   - Build/CI only -> GitHub Actions secrets
   - Runtime on Cloudflare Pages/Workers -> Cloudflare environment variables / secrets
   - Local development only -> `.env.example` placeholder only, never real values in repo
3. Remove any hardcoded secret usage from the codebase.
4. Update the code so it reads secrets from the correct environment variables in the correct runtime.
5. Make the repo deploy correctly after the migration.

Detailed tasks:
## Task 1: Audit
- Search the codebase for `process.env`, `import.meta.env`, `os.environ`, `dotenv`, `wrangler`, `cloudflare`, `github`, `secret`, `token`, `apiKey`, `API_KEY`, `AUTH`, `PASSWORD`, `PRIVATE_KEY`.
- List all discovered env vars and where they are used.
- Detect whether the app runs as:
  - static site on Cloudflare Pages,
  - Pages Functions / Workers,
  - GitHub Actions CI/CD,
  - local scripts only.

## Task 2: Decide destination for each secret
For each secret, produce a table:
- Secret name
- Used by
- Runtime or CI
- Destination: GitHub Secret / Cloudflare Secret / Cloudflare Pages env var / keep local only
- Why

Decision rules:
- If used only in GitHub Actions workflows, store in GitHub Actions secrets.
- If used by Cloudflare runtime code (Workers, Pages Functions), store in Cloudflare secrets/environment variables.
- If exposed to client-side browser code, do NOT treat as a secret; replace with safe public config if necessary.
- If a secret should never be shipped to the browser, refactor the code so it is used server-side only.

## Task 3: Refactor code
- Update the codebase to use the final env variable names consistently.
- Remove any dependency on Replit Secrets for production.
- Add safe fallbacks for local dev only.
- Create or update `.env.example` with placeholders only, no real values.
- Update any Cloudflare config (`wrangler.toml`, `wrangler.jsonc`, Pages config) and GitHub workflow files to reference the new variable names.

## Task 4: Deployment config
- If this project is hosted on Cloudflare Pages, configure runtime secrets the Cloudflare way.
- If there are GitHub Actions workflows, wire CI-only secrets through GitHub Actions secrets.
- Remove obsolete deployment paths and secret references.

## Task 5: Documentation
Update README / replit.md with:
- which secrets go to GitHub
- which secrets go to Cloudflare
- exact variable names
- where to set them in each dashboard
- what is required for local development
- what changed in the codebase

## Task 6: Safety checks
- Do not print secret values.
- Do not commit any real credentials.
- If any secret appears to already be committed, stop and tell me exactly which file and how to remediate.
- Show me a migration summary before committing.

## Task 7: Commit
After showing me the plan and changed files, create a single commit with a message like:
`chore: migrate secrets to GitHub and Cloudflare env configuration`

Important implementation guidance:
- Prefer Cloudflare for runtime secrets if the app executes there.
- Prefer GitHub Secrets only for CI/CD automation.
- Never leave secrets in frontend-exposed code.
- Preserve existing functionality.
- If uncertain whether a variable is secret or public, flag it and ask before finalizing.
